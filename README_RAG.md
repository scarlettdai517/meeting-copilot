# Smart RAG 策略设计说明

本文档说明本项目会议问答功能的 RAG 设计（`smart_rag.py`），重点解释：

- 为什么要做会议时长自适应（超短/短会/长会分流）
- 为什么长短会使用不同的 chunk 切分与检索策略
- BM25 + RRF、cross-encoder 重排、动态 Top-K 的具体机制
- 各项阈值/参数当前取值与设计理由

> 说明：测试数据、跑分命令、报告格式等实验操作说明，统一见  
> `data/benchmark/README_RAG_TEST.md`。  
> 本文不重复测试流程，只讲策略与设计决策。

---

## 1. 目标与场景特性

会议语音转写（ASR）类文本与普通文档不同，典型特点包括：

- 口语化强，存在重复、打断、省略、修正；
- 时间线长，关键信息可能跨段分布；
- 问题常是“汇总型、多跳型”而非单句事实查询；
- 同一关键词可能在不同语境反复出现，容易引入“伪相关”片段。

因此，本项目采用“分流 + 多阶段筛选”的 RAG 方案，而不是单一固定 top-k 语义检索。

---

## 2. 总体流程（Smart RAG）

Smart RAG 主流程如下：

1. 估算 transcript token 数并进行会议类型分类；
2. 按会议类型选择 chunk 切分参数；
3. 初始候选召回（语义检索，短会可叠加 BM25）；
4. 候选融合（RRF）；
5. 候选重排（cross-encoder，失败则回退启发式）；
6. 动态阈值 + 最大块数约束，选择最终上下文块；
7. 交给 LLM 生成回答（系统提示词固定）。

对应代码入口：

- 检索主函数：`smart_rag.py` 中 `retrieve_smart()`
- 应用问答入口：`smart_rag.py` 中 `answer_question()`
- 前端调用：`app.py` -> `from smart_rag import answer_question`

---

## 3. 会议时长自适应（超短/短会/长会）

### 3.1 分类阈值

当前阈值：

- 超短会：`token < 3000`（一般会议时长15分钟内）
- 短会：`3000 <= token < 10000`（一般会议时长在15-45分钟内）
- 长会：`token >= 10000`（一般会议时长大于45分钟）

对应参数：

- `ULTRASHORT_THRESHOLD = 3000`
- `LONG_THRESHOLD = 10000`

### 3.2 设计理由

- 超短会文本本身较短，直接全文给模型成本可控，且可避免检索环节漏召；
- 短会信息密度高、主题集中，适合更精细切分与关键词增强；
- 长会跨度大、信息分散，适合更大 chunk 以保留上下文完整性。

---

## 4. 分块策略（Chunking）

### 4.1 当前参数

- 短会：`chunk=256`，`overlap=50`
- 长会：`chunk=512`，`overlap=100`

对应参数：

- `SHORT_CHUNK_TOKENS = 256`
- `SHORT_OVERLAP_TOKENS = 50`
- `LONG_CHUNK_TOKENS = 512`
- `LONG_OVERLAP_TOKENS = 100`

### 4.2 切分方法

不是简单“硬截断”，而是先按语义单元构建再聚合：

- 优先保留时间戳边界（如 `[HH:MM:SS]`）；
- 在句号、问号、感叹号、分号等句边界处分句；
- 再按目标 token 大小聚合，并加 overlap。

这能兼顾：

- chunk 内语义相对完整；
- chunk 间边界信息不丢失；
- 与 GT chunk id 对齐时，优先复用已有 `chunks_*.json`。

---

## 5. 初始召回：语义检索 + BM25（按会议类型分流）

### 5.1 候选规模

- 短会候选数：`SHORT_CANDIDATES = 12`
- 长会候选数：`LONG_CANDIDATES = 15`

### 5.2 BM25 开关（分流）

当前开关：

- `SMART_V2_USE_BM25_FOR_SHORT = True`
- `SMART_V2_USE_BM25_FOR_LONG = False`
- `SMART_V2_SEMANTIC_ONLY = False`（总开关，若为 True 则长短都只走语义）

### 5.3 设计理由

- 短会常出现强结构化关键词（字段名、导出格式、角色称谓等），BM25 有助补召回；
- 长会关键词重复与漂移更明显，BM25 更容易引噪，因此默认关闭；
- 保留总开关用于实验：可快速切到“纯语义模式”做对照。

---

## 6. 候选融合：RRF（Reciprocal Rank Fusion）

当 BM25 开启时，语义榜单与 BM25 榜单通过 RRF 融合：

- 融合参数：`RRF_K = 60`
- 融合后去重并按分数排序，再截取候选上限。

设计意义：

- 不依赖不同检索器分数尺度统一（只依赖排名）；
- 能稳定融合“语义相关性”与“关键词命中性”；
- 对中文口语转写场景比直接加权原始分数更稳。

---

## 7. 重排：Cross-Encoder + 回退机制

### 7.1 当前设置

- 开关：`SMART_V2_ENABLE_RERANK = True`
- 默认模型：`BAAI/bge-reranker-v2-m3`

### 7.2 策略

- 优先使用 cross-encoder 对候选逐条打相关性分；
- 若依赖缺失或模型不可用，自动回退启发式重排（不中断流程）。

### 7.3 设计理由

初筛（语义/BM25）擅长“找得到”，但不擅长“排得准”；
重排阶段用于减少“看起来相关但实际无效”的噪声块，直接影响 precision。

---

## 8. 动态 Top-K：阈值自适应 + 上限约束

### 8.1 公式

对当前问题候选分数分布计算：

- `mu = 平均值`
- `sigma = 标准差`
- `threshold = max(mu - std_weight * sigma, abs_floor)`

并按排序从高到低选取：

- 直到分数低于 `threshold`，或达到 `max_chunks`；
- 若都低于阈值，至少保留 1 块。

### 8.2 当前参数

- 短会：
  - `SHORT_STD_WEIGHT = 0.2`
  - `SHORT_MAX_CHUNKS = 6`
- 长会：
  - `LONG_STD_WEIGHT = 0.4`
  - `LONG_MAX_CHUNKS = 6`
- 公共下限：
  - `DYNAMIC_ABS_FLOOR = 0.30`

### 8.3 参数含义与调参方向

在公式 `mu - std_weight*sigma` 下：

- `std_weight` 越大 -> 阈值越低 -> 选得更宽松（Recall 倾向上升，Precision 倾向下降）；
- `std_weight` 越小 -> 阈值越高 -> 选得更严格（Precision 倾向上升，Recall 倾向下降）；
- `max_chunks` 是最终噪声闸门，上调利于召回，下调利于精度。

### 8.4 为什么短会系数更小、长会系数更大

当前 `short=0.2`、`long=0.4` 的核心逻辑：

- 短会主题更集中，阈值可以更严格，减少无关片段；
- 长会信息更分散，阈值需相对放松，避免漏掉跨段证据。

---

## 9. 超短会策略（Smart 专属）

当 `token < 3000` 时，Smart RAG 直接全文：

- 不走检索，不走重排；
- 直接构造上下文并回答。

这样做的原因：

- 成本可控；
- 避免短文本被切块后引入不必要检索误差；
- 对超短会议问答稳定性通常更高。

---

## 10. 传统基线与 Smart 的关系

同一代码文件中同时保留了传统基线与 Smart 策略：

- `retrieve_traditional()`：固定语义 top-5，无动态阈值；
- `retrieve_smart()`：分流 + 混检/融合 + 重排 + 动态 Top-K。

这样设计的好处：

- 基线与升级策略共享同一套分块与数据对齐逻辑；
- 跑分对比时变量更可控，便于定位收益来源。
