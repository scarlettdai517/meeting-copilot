# Smart RAG - 智能检索增强生成系统

## 📌 设计理念

**核心目标**: 让RAG问答效果接近"直接把全文给GPT"，同时保留RAG的优势（可扩展性、成本控制、可追溯性）

**核心挑战**:
- 用户反馈：传统RAG检索不准，答案不如直接给GPT全文智能
- 原因分析：Chunk太小、检索不完整、Prompt限制性太强

**解决方案**: Smart RAG - 自适应的智能检索系统

---

## 🎯 Smart RAG vs 传统RAG

### 传统RAG的问题

```
传统RAG流程：
用户问题 → 固定chunk(250字) → 关键词/语义检索 → Top-6 chunks → 限制性Prompt → LLM

问题：
❌ Chunk太小：250字常常割裂上下文
❌ Top-K固定：总是6个chunks，可能信息不足或冗余
❌ Prompt太严格："只基于证据回答"限制了LLM推理能力
❌ 一刀切：短会议也切chunk，浪费效率
```

### Smart RAG的创新

```
Smart RAG流程：
用户问题 → 自适应策略 → 优化检索 → 智能Prompt → LLM
             ↓
      短会议（<3000字）→ 直接全文
      长会议（≥3000字）→ 大chunk(600字) + 动态Top-K + 覆盖率检测

优势：
✅ 自适应：短会议直接全文，长会议才用RAG
✅ 大chunk：600字保留完整上下文
✅ 动态Top-K：根据相似度和覆盖率智能调整
✅ 宽松Prompt：鼓励LLM综合推理，不强制限制
```

---

## 🏗️ 核心技术实现

### 1. 自适应策略选择

**问题**: 短会议为什么要切chunk？直接给全文不是更好？

**方案**: 根据会议长度自动选择策略

```python
def smart_retrieve(question, transcript, cache_id=None):
    transcript_len = len(transcript)

    # 策略1: 短会议（<3000字）→ 直接全文
    if transcript_len < 3000:
        return ([{"id": 0, "text": transcript}], "full_text")

    # 策略2: 长会议（≥3000字）→ 优化RAG
    else:
        return (optimized_rag_retrieve(...), "rag")
```

**为什么3000字？**
- 3000字 ≈ 2000 tokens
- GPT-4o-mini上下文窗口128K tokens，2000 tokens只占1.5%
- 完全在处理范围内，无需切分
- 超过3000字才考虑使用RAG节省成本和提高精度

**效果**:
```
会议1（1500字）：
  传统RAG: 切成6个chunks → 可能丢失上下文连贯性
  Smart RAG: 直接全文 → 100%完整信息 ✅

会议2（15000字）：
  传统RAG: 60个chunks，取6个 → 覆盖率10%
  Smart RAG: 25个大chunks，动态取8-15个 → 覆盖率30-60% ✅
```

### 2. 优化的Chunk策略

**问题**: 250字太小，经常割裂一个完整的段落

**方案**: 增大Chunk + 增大重叠

```python
# 传统RAG
CHUNK_SIZE = 250  # 太小
CHUNK_OVERLAP = 50

# Smart RAG
CHUNK_SIZE = 600  # 2.4倍增大，能容纳2-3个完整段落
CHUNK_OVERLAP = 100  # 2倍增大，避免信息在边界丢失
```

**实际效果对比**:

```
原文：
"张三负责前端开发，目前完成了70%，预计下周五完成。
前端包括登录页面、仪表盘、数据可视化三个模块。
其中数据可视化是难点，可能需要额外2天。"

传统RAG（250字）：
Chunk 1: "张三负责前端开发，目前完成了70%，预计下周五完成。
         前端包括登录页面、仪表盘、数据可..."
Chunk 2: "...可视化三个模块。其中数据可视化是难点，可能需要额外2天。"
❌ "数据可视化"被割裂成两个chunk

Smart RAG（600字）：
Chunk 1: "张三负责前端开发，目前完成了70%，预计下周五完成。
         前端包括登录页面、仪表盘、数据可视化三个模块。
         其中数据可视化是难点，可能需要额外2天。"
✅ 完整保留上下文
```

### 3. 动态Top-K选择

**问题**: 固定返回6个chunks，可能太少或太多

**方案**: 根据相似度阈值 + 覆盖率动态调整

```python
# 阶段1: 初步检索Top-12
similarities = compute_similarities(question, chunks)
top_12 = similarities[:12]

# 阶段2: 相似度过滤
selected = []
for sim, chunk in top_12:
    if sim > 0.3 or len(selected) < 4:  # 相似度阈值 or 最少4个
        selected.append(chunk)

# 阶段3: 覆盖率检测
coverage = sum(len(c['text']) for c in selected) / len(transcript)

if coverage < 0.3:  # 覆盖率太低，补充更多
    selected = [chunk for _, chunk in similarities[:15]]
```

**效果**:

| 场景 | 传统RAG | Smart RAG |
|------|---------|-----------|
| **高度相关** | 固定6个 | 4-6个（够了就不多拿） |
| **部分相关** | 固定6个 | 8-10个（保证覆盖率） |
| **低相关性** | 固定6个（可能都不相关） | 10-15个（宁可多不可少） |

---

### 4. 智能Prompt设计

**问题**: 传统Prompt太限制LLM，"只能基于证据回答"抑制了推理能力

**传统RAG Prompt**:
```
Rules:
- Answer ONLY using the evidence blocks below.
- If the evidence is insufficient, say you don't know.
- Keep the answer concise and actionable.

❌ 太严格！即使chunks包含相关信息，LLM可能因为"不确定"而拒绝回答
```

**Smart RAG Prompt**:
```
你是一个专业的会议助手。请基于以下会议片段回答用户的问题。

会议相关片段（按相关度排序）：
[片段内容...]

回答要求：
1. 基于上述片段提供准确、有用的回答
2. 如果片段中信息充足，直接回答
3. 如果片段中信息不足，说明需要更多信息
4. 尽可能引用片段中的关键信息（例如："根据片段X，..."）

✅ 更自然！鼓励LLM综合推理，但要求引用证据
```

**关键改进**:
- ❌ 删除"ONLY"限制词
- ✅ 改为"基于...提供准确、有用的回答"
- ✅ 强调"尽可能引用"而不是"必须引用"
- ✅ 允许LLM说"信息不足"而不是强制"我不知道"

---

### 5. 智能回退机制

**问题**: 如果语义检索失败怎么办？

**方案**: 多层回退保障

```python
try:
    # 尝试语义检索
    chunks_embeddings = prepare_chunks_for_retrieval(chunks, cache_id)
    selected = semantic_retrieve(question, chunks, chunks_embeddings)

except Exception as e:
    # 回退：使用全文
    return ([{"id": 0, "text": transcript}], "full_text")
```

**回退策略**:
1. 首选：语义检索（最准确）
2. 回退：直接全文（保证可用）
3. 永远不会失败（总能给出答案）

## 🎓 为什么Smart RAG更好？

### 1. 自适应 = 智能

传统RAG：
```
无论会议长短，一律切chunk检索
→ 短会议浪费效率，长会议信息不足
```

Smart RAG:
```
短会议 → 全文（最简单最有效）
长会议 → 大chunk + 动态Top-K（兼顾质量和成本）
→ 因地制宜，智能选择
```

---

### 2. 大Chunk = 完整上下文

传统RAG：
```
250字/chunk → 割裂段落 → 丢失上下文
例: "张三负责前端" 和 "需要额外2天" 在不同chunk
→ LLM看不到完整信息
```

Smart RAG:
```
600字/chunk → 保留2-3个完整段落 → 上下文完整
例: "张三负责前端开发，包括X、Y、Z，目前70%，预计下周五完成，
     但数据可视化复杂，需要额外2天"
→ LLM能综合理解
```

---

### 3. 动态Top-K = 按需检索

传统RAG：
```
固定Top-6:
- 问题简单 → 6个太多，有噪音
- 问题复杂 → 6个太少，信息不足
```

Smart RAG:
```
相似度高 → 4-6个（够了）
相似度中 → 8-10个（补充）
相似度低 → 10-15个（宁多不少）
+ 覆盖率检测（<30%自动补充）
→ 自适应调整，保证信息充足
```

---

### 4. 智能Prompt = 释放LLM能力

传统RAG：
```
"ONLY use evidence, say you don't know if uncertain"
→ 过度限制，即使chunks有相关信息，LLM也可能说"不知道"
```

Smart RAG:
```
"基于片段提供准确、有用的回答，尽可能引用"
→ 鼓励推理，但要求有据，平衡准确性和完整性
```

---

### 5. 回退机制 = 可靠性

传统RAG：
```
语义检索失败 → 降级到关键词检索 → 可能仍然检索不准
→ 答案质量下降
```

Smart RAG:
```
语义检索失败 → 直接用全文
→ 保证答案质量，永远可用
```

## 🔬 技术细节

### 语义检索原理

```python
# 1. 将问题和chunks转换为向量
question_embedding = get_embedding("谁负责前端？")
# → [0.12, -0.45, 0.78, ..., 0.23]  (1536维)

chunk_embeddings = [get_embedding(chunk['text']) for chunk in chunks]
# → [[0.14, -0.42, ...], [0.05, -0.33, ...], ...]

# 2. 计算余弦相似度
similarities = [
    cosine_similarity(question_embedding, chunk_emb)
    for chunk_emb in chunk_embeddings
]
# → [0.87, 0.23, 0.91, 0.15, ...]

# 3. 排序并选择Top-K
sorted_indices = argsort(similarities, reverse=True)
selected = [chunks[i] for i in sorted_indices[:top_k]]
```

**为什么Embedding能理解语义？**

```
"前端工程师" → Embedding → [0.12, -0.45, 0.78, ...]
"前端开发"   → Embedding → [0.14, -0.42, 0.80, ...]
"UI负责人"   → Embedding → [0.15, -0.43, 0.79, ...]

余弦相似度:
- "前端工程师" vs "前端开发": 0.95（很相似！）
- "前端工程师" vs "后端API": 0.35（不太相关）
```

Embedding模型在训练时学习了语义关系，所以能理解：
- 同义词："工程师" ≈ "开发"
- 领域知识："UI" ≈ "前端"
- 语义相关："负责" ≈ "汇报"

### 缓存机制

```python
# 首次：生成embeddings（需要API调用）
chunks_embeddings = prepare_chunks_for_retrieval(
    chunks,
    cache_id="meeting_abc123",
    use_cache=True
)
# → 调用OpenAI API，耗时~2秒
# → 保存到 data/embeddings_cache/meeting_abc123.json

# 后续：从缓存加载（秒级）
chunks_embeddings = prepare_chunks_for_retrieval(
    chunks,
    cache_id="meeting_abc123",
    use_cache=True
)
# → 从缓存读取，耗时~0.01秒 ✅
```

**缓存收益**:
- 首次问答：慢（~2秒生成embeddings）
- 后续问答：快（直接用缓存）
- 成本降低：只需一次Embedding API调用

## 🚀 使用指南

### 配置

在 `.env` 文件中设置:
```bash
OPENAI_API_KEY=sk-your-key-here
OPENAI_BASE_URL=https://api.openai.com/v1  # 可选
```

```bash
streamlit run app.py
```

### 使用

1. 输入会议转写并Generate
2. 在问答区输入问题
3. 点击Ask
4. 查看智能回答 + 检索详情

### 调试

展开"检索详情"可以看到：
- 使用的策略（全文/RAG）
- 原因说明
- 检索到的chunks数量
- 覆盖率和相似度
- 具体检索到的内容片段
